# 글 제목 전략 및 하이퍼링크 매칭 개선 방안

## 현재 문제점

1. **제목 구조의 모호성**
   - 짧은 제목: "Next.js" (명확하지만 정보 부족)
   - 문장형 제목: "Next.js의 장점" (정보는 풍부하지만 매칭 어려움)

2. **하이퍼링크 매칭의 한계**
   - "Next.js의 장점"이라는 글에서 "Next.js"만 언급하면 매칭 가능
   - 하지만 "Next.js"라는 별도 글이 있으면 충돌 가능
   - 문장형 제목 전체를 매칭하려면 정확히 같은 문장이 필요

## 해결 방안

### 방안 1: 별칭(Alias) 시스템 (추천)

**개념:**
- 각 글에 매칭에 사용할 별칭(키워드) 목록을 저장
- 제목은 자유롭게 문장형으로 작성 가능
- 별칭을 통해 다양한 표현으로 링크 생성 가능

**예시:**
```
제목: "Next.js의 장점"
별칭: ["Next.js", "NextJS", "next.js", "Next.js 프레임워크"]

→ 다른 글에서 "Next.js" 또는 "NextJS"를 언급하면 자동 링크 생성
```

**장점:**
- 제목은 자유롭게 문장형 사용 가능
- 다양한 표현으로 링크 생성 가능
- 핵심 키워드와 제목을 분리하여 관리

**구현:**
- `Article` 모델에 `aliases` 필드 추가 (JSON 배열)
- `link-detector.ts`에서 별칭도 함께 매칭
- 제목과 별칭 중 가장 긴 매칭을 우선 사용

### 방안 2: 제목 구조화

**개념:**
- 제목을 "핵심 키워드 | 부제목" 형식으로 구조화
- 예: "Next.js | 프레임워크의 장점과 특징"

**장점:**
- 구조가 명확함
- 핵심 키워드 추출이 쉬움

**단점:**
- 제목 형식이 제한됨
- 사용자가 규칙을 기억해야 함

### 방안 3: 자동 키워드 추출

**개념:**
- 제목에서 핵심 키워드를 자동으로 추출
- 예: "Next.js의 장점" → "Next.js" 추출

**장점:**
- 사용자 개입 최소화

**단점:**
- 추출 정확도 문제
- 한글 제목의 경우 추출이 어려움

## 추천 구현: 별칭 시스템

### 1. 데이터베이스 스키마 변경

```prisma
model Article {
  // ... 기존 필드
  aliases String[] @default([]) // 매칭에 사용할 별칭 목록
}
```

### 2. 링크 감지 로직 개선

```typescript
// 제목과 별칭 모두를 매칭 대상으로 사용
// 긴 매칭부터 우선 처리 (더 구체적인 매칭 우선)
```

### 3. 사용자 가이드

- 제목: 자유롭게 문장형으로 작성 가능
- 별칭: 다른 글에서 언급할 수 있는 키워드들 (선택사항)
- 별칭 미지정 시: 제목 전체가 매칭 키워드로 사용됨

## 예시 시나리오

### 시나리오 1: 기본 개념 글
```
제목: "Next.js"
별칭: [] (또는 ["NextJS", "next.js"])

→ 다른 글에서 "Next.js"를 언급하면 링크 생성
```

### 시나리오 2: 특정 주제 글
```
제목: "Next.js의 장점"
별칭: ["Next.js", "NextJS"]

→ 다른 글에서 "Next.js"를 언급하면 이 글로 링크 생성
→ "Next.js"라는 기본 개념 글이 있으면 충돌 가능
```

### 시나리오 3: 충돌 해결
```
글 A: 제목 "Next.js", 별칭: []
글 B: 제목 "Next.js의 장점", 별칭: ["Next.js"]

→ "Next.js" 언급 시:
  - "Next.js" 정확히 매칭 → 글 A 우선
  - "Next.js의 장점" 언급 → 글 B 우선
  - 별칭으로 매칭 → 글 B도 가능
```

## 구현 우선순위

1. **1단계**: 별칭 필드 추가 및 기본 매칭 로직 구현
2. **2단계**: 에디터에서 별칭 입력 UI 추가
3. **3단계**: 자동 별칭 제안 기능 (선택사항)
